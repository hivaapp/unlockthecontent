## Part 2 — Row Level Security Policies

---

### What This Part Does

Enables RLS on all 27 tables. Writes explicit policies for every table — who can SELECT, INSERT, UPDATE, DELETE. No table is left with implicit access. Every policy is named clearly so it can be audited, modified, or dropped individually. All policies run as `SECURITY DEFINER` where elevation is needed, `SECURITY INVOKER` everywhere else.

The guiding rules before writing a single policy:

- **Public data** — explore resources, creator profiles, link pages. Anyone can read. Only owners can write.
- **Private data** — chat messages, earnings, subscribers, analytics. Only the directly involved parties can read or write.
- **Pairing chat** — the strictest policy in the system. Only the two matched participants can read their session's messages. Not even the creator whose campaign it is.
- **Admin data** — reports, trust events. Users can insert and read their own. No user can modify after submission.
- **Service role** — Supabase Edge Functions and backend operations use the service role key which bypasses RLS entirely. All automated operations (scheduled messages, session expiry, trust score events) run as service role.

### Block 2 — Helper Functions Used By Policies

```sql
-- Returns true if the current user is blocked by OR has blocked the given user.
-- Used to prevent blocked users from seeing each other's content.
CREATE OR REPLACE FUNCTION is_blocked_relationship(other_user_id UUID)
RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.blocked_users
    WHERE (blocker_id = auth.uid() AND blocked_id = other_user_id)
       OR (blocker_id = other_user_id AND blocked_id = auth.uid())
  );
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Returns true if the current user is a participant in the given pairing session.
CREATE OR REPLACE FUNCTION is_session_participant(p_session_id UUID)
RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1
    FROM public.pairing_sessions ps
    JOIN public.pairing_participants pp_a ON pp_a.id = ps.participant_a_id
    JOIN public.pairing_participants pp_b ON pp_b.id = ps.participant_b_id
    WHERE ps.id = p_session_id
      AND (pp_a.user_id = auth.uid() OR pp_b.user_id = auth.uid())
  );
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Returns true if the current user is a participant in the given pairing session
-- identified by participant record.
CREATE OR REPLACE FUNCTION is_participant_in_session(p_participant_id UUID)
RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1
    FROM public.pairing_participants pp
    WHERE pp.id = p_participant_id
      AND pp.user_id = auth.uid()
  );
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Returns true if the current user is a participant in a direct conversation.
CREATE OR REPLACE FUNCTION is_conversation_participant(p_conversation_id UUID)
RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.direct_conversations
    WHERE id = p_conversation_id
      AND (participant_a_id = auth.uid() OR participant_b_id = auth.uid())
  );
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Returns the creator_id for a given link.
CREATE OR REPLACE FUNCTION get_link_creator(p_link_id UUID)
RETURNS UUID AS $$
  SELECT creator_id FROM public.links WHERE id = p_link_id;
$$ LANGUAGE sql SECURITY DEFINER STABLE;
```

### Block 3 — Users Table Policies

```sql
-- READ: Any authenticated user can read public profile fields.
-- Blocked relationships are handled at the application query level, not here,
-- because the profile page itself still needs to load for the blocker to manage the block.
CREATE POLICY "users_select_public"
  ON public.users FOR SELECT
  USING (true);
-- Note: sensitive fields (stripe_account_id, referred_by) are excluded
-- at the query level in the frontend — never SELECT * from users in client code.

-- INSERT: Handled entirely by the handle_new_user() trigger.
-- No direct INSERT allowed from client.
CREATE POLICY "users_insert_trigger_only"
  ON public.users FOR INSERT
  WITH CHECK (false);
-- The trigger runs as SECURITY DEFINER so it bypasses this policy.

-- UPDATE: Users can only update their own row.
-- Sensitive fields (is_pro, is_verified, trust_score, stripe_account_id)
-- are updated only via Edge Functions running as service role — never directly from client.
CREATE POLICY "users_update_own"
  ON public.users FOR UPDATE
  USING (auth.uid() = id)
  WITH CHECK (
    auth.uid() = id
    -- Prevent client from elevating their own privileges.
    -- These fields can only be changed by service role (Edge Functions / webhooks).
    -- Enforced here as an extra layer even though service role bypasses RLS.
    AND is_pro = (SELECT is_pro FROM public.users WHERE id = auth.uid())
    AND is_verified = (SELECT is_verified FROM public.users WHERE id = auth.uid())
    AND trust_score = (SELECT trust_score FROM public.users WHERE id = auth.uid())
    AND stripe_connected = (SELECT stripe_connected FROM public.users WHERE id = auth.uid())
  );

-- DELETE: Not allowed from client. Account deletion is an Edge Function operation.
CREATE POLICY "users_delete_never_client"
  ON public.users FOR DELETE
  USING (false);
```

### Block 5 — Blocked Users Policies

```sql
-- READ: A user can only see their own block relationships.
-- They cannot see who has blocked them — only who they have blocked.
CREATE POLICY "blocked_users_select_own"
  ON public.blocked_users FOR SELECT
  USING (auth.uid() = blocker_id);

-- INSERT: A user can only create blocks where they are the blocker.
CREATE POLICY "blocked_users_insert_own"
  ON public.blocked_users FOR INSERT
  WITH CHECK (auth.uid() = blocker_id AND auth.uid() != blocked_id);

-- DELETE: A user can only remove blocks they created.
CREATE POLICY "blocked_users_delete_own"
  ON public.blocked_users FOR DELETE
  USING (auth.uid() = blocker_id);

-- No UPDATE policy — blocks are created or deleted, never edited.
```

### Block 6 — Files Policies

```sql
-- READ: File metadata is readable by:
-- 1. The file owner always.
-- 2. Any authenticated user IF the file is attached to an active public link.
--    (Checked via the links table — if the link is active and the file is its content file.)
-- 3. Pairing participants can read files attached to their session's messages.
-- Completion asset files are only readable after session completion — enforced at Edge Function level,
-- not RLS, because RLS cannot efficiently check session completion state per-request.
CREATE POLICY "files_select_owner"
  ON public.files FOR SELECT
  USING (
    auth.uid() = owner_id
    OR EXISTS (
      SELECT 1 FROM public.links
      WHERE file_id = files.id AND is_active = true
    )
    OR EXISTS (
      SELECT 1 FROM public.pairing_messages pm
      WHERE pm.file_id = files.id
        AND is_session_participant(pm.session_id)
    )
    OR EXISTS (
      SELECT 1 FROM public.direct_messages dm
      WHERE dm.file_id = files.id
        AND is_conversation_participant(dm.conversation_id)
    )
  );

-- INSERT: Users can only upload files they own.
CREATE POLICY "files_insert_own"
  ON public.files FOR INSERT
  WITH CHECK (auth.uid() = owner_id);

-- UPDATE: File owners can update metadata (e.g. renaming).
-- R2 key cannot be changed after creation — enforced at application level.
CREATE POLICY "files_update_own"
  ON public.files FOR UPDATE
  USING (auth.uid() = owner_id)
  WITH CHECK (auth.uid() = owner_id);

-- DELETE: File owners can delete file records.
-- Actual R2 object deletion is handled by an Edge Function after this.
CREATE POLICY "files_delete_own"
  ON public.files FOR DELETE
  USING (auth.uid() = owner_id);
```

-- ── PAIRING PARTICIPANTS ─────────────────────────────────────────────────

-- READ: 
-- The creator sees all participants in their campaigns (for analytics).
-- A participant sees their own record.
-- A participant sees their matched partner's record (after matching).
-- Nobody else.
CREATE POLICY "pairing_participants_select"
  ON public.pairing_participants FOR SELECT
  USING (
    -- Own record
    auth.uid() = user_id
    OR
    -- Creator of the campaign
    auth.uid() = get_link_creator(link_id)

      SELECT 1 FROM public.pairing_sessions ps

        (ps.participant_a_id = pairing_participants.id AND is_participant_in_session(ps.participant_b_id))
        OR
        (ps.participant_b_id = pairing_participants.id AND is_participant_in_session(ps.participant_a_id))
      )
    )
  );

    auth.uid() = user_id
    AND EXISTS (

      FROM public.pairing_configs pc
      WHERE pc.link_id = pairing_participants.link_id

CREATE POLICY "pairing_participants_update_own"
  ON public.pairing_participants FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "pairing_participants_delete_own"
  ON public.pairing_participants FOR DELETE
  USING (auth.uid() = user_id);

-- ── PAIRING MESSAGES ────────────────────────────────────────────────────
-- THE MOST CRITICAL POLICY IN THE SYSTEM.
-- Pairing messages are strictly private to the two participants.
-- The creator of the campaign CANNOT read private messages.
-- Only broadcast messages (is_broadcast = true) are readable by the creator.

-- READ:
-- Private messages: only the two session participants.
-- Broadcast messages: session participants AND the campaign creator.
CREATE POLICY "pairing_messages_select"
  ON public.pairing_messages FOR SELECT
  USING (
    -- Always: session participants can read all messages in their session.
    is_session_participant(session_id)
    OR
    -- Broadcast messages only: creator can see what they sent (for broadcast history).
    (
      is_broadcast = true
      AND auth.uid() = (
        SELECT l.creator_id
        FROM public.pairing_sessions ps
        JOIN public.links l ON l.id = ps.link_id
        WHERE ps.id = pairing_messages.session_id
      )
    )
  );

-- INSERT: Session participants can send messages in their session.
-- Broadcast messages are inserted by Edge Function (service role) — client sends
-- to a separate broadcast endpoint, not directly to this table.
CREATE POLICY "pairing_messages_insert_participant"
  ON public.pairing_messages FOR INSERT
  WITH CHECK (
    auth.uid() = sender_id
    AND is_broadcast = false  -- broadcasts only via service role
    AND is_session_participant(session_id)
    AND EXISTS (
      SELECT 1 FROM public.pairing_sessions
      WHERE id = pairing_messages.session_id
        AND status = 'active'  -- cannot message in expired/completed sessions
    )
  );

-- UPDATE: Only the sender can update their own messages (for reactions, edit, soft delete).
CREATE POLICY "pairing_messages_update_own"
  ON public.pairing_messages FOR UPDATE
  USING (
    auth.uid() = sender_id
    AND is_session_participant(session_id)
  )
  WITH CHECK (
    auth.uid() = sender_id
    AND is_broadcast = false  -- sender cannot change a broadcast to private or vice versa
  );

-- DELETE: No hard deletes — soft delete via is_deleted = true flag.

### Block 11 — Direct Messaging Policies

```sql
-- ── MESSAGE REQUESTS ────────────────────────────────────────────────────

-- READ: Sender sees their sent requests. Recipient sees incoming requests.
CREATE POLICY "message_requests_select"
  ON public.message_requests FOR SELECT
  USING (
    auth.uid() = sender_id
    OR auth.uid() = recipient_id
  );

-- INSERT: Any authenticated user can send a request.
-- UNIQUE(sender_id, recipient_id) constraint prevents duplicates.
-- Application validates: not blocked, not self, recipient exists.
CREATE POLICY "message_requests_insert_auth"
  ON public.message_requests FOR INSERT
  WITH CHECK (
    auth.uid() = sender_id
    AND auth.uid() != recipient_id
    -- Sender is not blocked by recipient
    AND NOT EXISTS (
      SELECT 1 FROM public.blocked_users
      WHERE blocker_id = recipient_id AND blocked_id = auth.uid()
    )
    -- Sender has not blocked recipient
    AND NOT EXISTS (
      SELECT 1 FROM public.blocked_users
      WHERE blocker_id = auth.uid() AND blocked_id = recipient_id
    )
  );

-- UPDATE: Only the recipient can update status (approve/decline).
-- Sender cannot retract after sending — they can only wait.
CREATE POLICY "message_requests_update_recipient"
  ON public.message_requests FOR UPDATE
  USING (auth.uid() = recipient_id)
  WITH CHECK (
    auth.uid() = recipient_id
    -- Status can only move forward: pending → approved OR pending → declined.
    -- Cannot un-decline or un-approve.
    AND status IN ('approved', 'declined')
  );

-- No DELETE. Historical record of all requests.


-- ── DIRECT CONVERSATIONS ─────────────────────────────────────────────────

-- READ: Only the two participants can see the conversation.
CREATE POLICY "direct_conversations_select"
  ON public.direct_conversations FOR SELECT
  USING (
    auth.uid() = participant_a_id
    OR auth.uid() = participant_b_id
  );

-- INSERT: Only service role (created by the request approval Edge Function).
-- Client approves a request; the Edge Function creates the conversation.
CREATE POLICY "direct_conversations_insert_service_only"
  ON public.direct_conversations FOR INSERT
  WITH CHECK (false);

-- UPDATE: Participants can update their own unread count (mark as read).
-- last_message fields updated by trigger (service role function context).
CREATE POLICY "direct_conversations_update_participant"
  ON public.direct_conversations FOR UPDATE
  USING (
    auth.uid() = participant_a_id
    OR auth.uid() = participant_b_id
  )
  WITH CHECK (
    -- Participants can only update unread counts, not change participants.
    participant_a_id = (SELECT participant_a_id FROM public.direct_conversations WHERE id = direct_conversations.id)
    AND participant_b_id = (SELECT participant_b_id FROM public.direct_conversations WHERE id = direct_conversations.id)
  );

-- No DELETE. Blocking a user hides conversation at app layer, not DB layer.


-- ── DIRECT MESSAGES ─────────────────────────────────────────────────────

-- READ: Only participants in the conversation.
CREATE POLICY "direct_messages_select"
  ON public.direct_messages FOR SELECT
  USING (is_conversation_participant(conversation_id));

-- INSERT: Participants can send messages. Soft-deleted messages cannot be replied to
-- (application layer validation). Blocked users cannot message (enforced at request
-- approval stage — if conversation exists, block cannot be active).
CREATE POLICY "direct_messages_insert_participant"
  ON public.direct_messages FOR INSERT
  WITH CHECK (
    auth.uid() = sender_id
    AND is_conversation_participant(conversation_id)
  );

-- UPDATE: Sender can soft-delete or add reactions to their own messages.
-- Recipients can mark as read and add reactions.
CREATE POLICY "direct_messages_update"
  ON public.direct_messages FOR UPDATE
  USING (is_conversation_participant(conversation_id))
  WITH CHECK (
    -- Only the sender can change content-level fields.
    -- Both participants can update is_read and reactions.
    is_conversation_participant(conversation_id)
  );

-- No hard DELETE.
```
